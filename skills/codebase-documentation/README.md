
## 文档输出两种范式对比分析

### 范式逻辑说明
- A: 对话先并行subagent做代码结构/细节探索（Evidence Packet），然后著对话进行串行文档输出
- B: 对话先串行分析代码（Research Notes），然后并行subagent各取所需输出文档

### 结论：方案 A 更科学，不需要改为方案 B

---
- 方案 A：并行探索 → 主对话串行写作（你的方案）

    - 并行 subagents（evidence-only）:
        Agent1: 探索子系统 A → Evidence Packet A
        Agent2: 探索子系统 B → Evidence Packet B
        Agent3: dependency check → Evidence Packet C

        主对话（串行，深度优先）:
        [读 Evidence Packets] → [写 Doc 1] → [写 Doc 2] → ...

    - 优势：
        1. 写作者拥有完整上下文：主对话持有所有 Evidence Packets + 全部已写文档的记忆，跨子系统关联不会丢失
        2. 一致性有保障：同一个 agent 写所有 sibling docs，shared anchors 的 ownership 自然一致，不会出现两个 agent 对同一机制描述矛盾
        3. 探索天然适合并行：不同子系统的代码阅读彼此独立，不需要共享上下文
        4. 深度瓶颈在正确的位置：写作需要深度思考（正确位置串行），探索是机械性工作（正确位置并行）
        5. 中间产物（Evidence Packet）是结构化的：比 Research Notes 更适合被主对话消费，因为它已经按 trace/L3/Key Files 组织好了

    - 劣势：
        1. 速度受限于主对话的串行写作
        2. 主对话 context window 随着文档数量增加会变紧

- 方案 B：串行研究 → 并行写作（我的方案）

    - Research Agent（串行）:
        [深度探索整个模块] → research_notes.md

        并行 Writer Agents:
        Agent1: [读 notes] + [写 Doc 1]
        Agent2: [读 notes] + [写 Doc 2]
        Agent3: [读 notes] + [写 Doc 3]

    - 优势：
        1. 写作可以并行，整体速度更快
        2. 每个 writer 不需要重复探索

    - 劣势：
        1. 跨文档一致性无法保证：Agent1 和 Agent2 同时写，shared anchor 可能被两个 agent 各自解释一遍且不一致
        2. Research Notes 是信息瓶颈：notes 必须足够详尽才能让 writer 不看源码就能写出有深度的文档，但足够详尽的 notes 本身就占大量 context
        3. writer 缺乏 sibling context：Agent1 不知道 Agent2 写了什么，无法做 cross-reference
        4. 深度在错误的位置被牺牲：写作（需要深度的地方）被分散到受限的 subagent，而探索（不那么需要深度的地方）却集中到了一个串行 agent

---
- 为什么方案 A 更科学

 核心论点：文档的深度和一致性来自写作者的上下文广度，而不是写作的并行度。

 1. 写作是上下文密集型任务：写一个子系统的 L2 数据流时，经常需要引用其他子系统的类——这要求写作者"记住"整个模块的结构。主对话天然拥有这个上下文。
 2. 探索是上下文稀疏型任务：阅读子系统 A 的代码不需要知道子系统 B 的代码。并行探索不丢失信息。
 3. Evidence Packet 是比 Research Notes 更优的中间格式：它已经按 skill 的覆盖链结构（trace → L3 → Key Files）组织，主对话可以直接缝合，不需要二次结构化。
 4. 你的 Subagent Orchestration Rules 和 Attention Dilution Signals 本质上是在保护写作深度：限制 fan-out、强制 evidence-only、检测 function names
 消失等——这些都是方案 A 的配套防护。
