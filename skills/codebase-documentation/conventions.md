# Document Output Conventions

> Reference file for file placement, naming, and cross-referencing. For document templates, see [templates.md](templates.md).

## Document Output Structure

All generated documentation MUST follow this directory layout. The goal is strict **separation of concerns**: narrative analysis, diagrams, and presentation materials each have their own home.

### Top-Level Layout

```
Document/
├── MasterIndex.md                 # Master index — the single navigation hub
├── <ConventionDocs>.md            # Convention / standard docs (optional, project-specific)
│
├── NNN-<TopicName>/               # Numbered topic folders (analysis markdown)
│   ├── NNN_<Module>_Analysis.md   # File numbers globally unique, within folder's allocated range
│   └── ...
│
├── diagrams/                      # ALL Excalidraw diagrams (grouped by domain)
│   ├── Global/                    # Global architecture diagrams
│   │   ├── Project_Architecture.excalidraw
│   │   └── Project_Architecture.png
│   ├── <DomainLabel>/
│   │   ├── *.excalidraw           # Editable source files
│   │   └── *.png                  # Exported viewable files (generated by export script)
│   └── ...
│
└── Presentation/                  # (Optional) Presentation-oriented materials
    ├── *.md
    └── diagrams/
        └── Dnn-<Description>.excalidraw  # Presentation-specific diagrams with order prefix
```

### Directory Naming Rules

| Level | Pattern | Example |
|-------|---------|---------|
| Topic folder | `NNN-<TopicName>` (three-digit zero-padded, decade-aligned) | `000-Overview`, `010-Framework`, `100-FeatureWorld` |
| Diagram subfolder | Short domain label (no number prefix) | `CoreModuleA`, `Infrastructure` |
| Global diagrams | Always `Global/` | `diagrams/Global/Project_Architecture.png` |
| Presentation diagram | `Dnn-<Description>` or `nn-<Description>` | `D01-SystemOverview`, `03-DataFlow` |

**Topic folder numbering** uses a decade-aligned scheme: each folder number is a multiple of 10 (e.g., `000`, `010`, `020`, ..., `090`, `100`, `110`). Reading order goes from general overview first, then subsystems by importance, finally utilities/tools. Three-digit zero-padding ensures correct lexicographic sorting even beyond `099`.

### Decade-Aligned Numbering & Variable Spacing

**Core rule**: folder numbers are multiples of 10; file numbers fall within the folder's allocated range, making every file number globally unique across the entire documentation set.

**Variable spacing** allows allocating wider ranges to folders that need more files:

| Folder Size | Spacing | File Slots | When to Use |
|-------------|---------|------------|-------------|
| Small (≤5 files expected) | ×10 | `N01–N09` (9 slots) | Utility modules, simple modules |
| Medium (6–15 files expected) | ×20 | `N01–N+19` (19 slots) | Large modules with sub-module splits |
| Large (>15 files expected) | ×30 | `N01–N+29` (29 slots) | Very large modules (e.g., 1000+ source files) |

**Example allocation**:

```
000-ProjectOverview/        → 001–009  (×10)
010-Framework/              → 011–029  (×20, large module with sub-splits)
030-BattleSystem/           → 031–049  (×20)
050-GameLogic/              → 051–079  (×30, very large module)
080-GameRuntime/            → 081–109  (×30)
110-Supporting/             → 111–119  (×10)
120-Extensions/             → 121–129  (×10)
130-FeatureWorldCity/       → 131–149  (×20)
150-FeatureBattle/          → 151–169  (×20)
170-FeatureOther/           → 171–189  (×20)
```

**Allocation rules**:

1. Spacing is determined during the MasterIndex Documentation Execution Plan phase, based on estimated file counts from the module inventory
2. Each folder's allocated range MUST NOT overlap with the next folder's start number
3. The Documentation Execution Plan table MUST include a "Range" column showing each folder's allocated file number range
4. If a folder exhausts its range during analysis, treat it as a signal to split the folder rather than breaking into the next range

### File Naming Rules

| File Type | Pattern | Example |
|-----------|---------|---------|
| Module analysis doc | `NNN_<ModuleName>_Analysis.md` | `011_Auth_Framework_Analysis.md` |
| Multi-module overview | `NNN_<TopicName>_Overview.md` | `031_BattleSystem_Overview.md` |
| End-to-end flow doc | `NNN_<FlowName>_E2EFlow.md` | `012_OrderFlow_E2EFlow.md` |
| Specialized analysis | `NNN_<Topic>_Analysis.md` | `013_CacheEviction_Analysis.md` |
| Architecture diagram | `<SystemName>_Architecture.excalidraw` | `Payment_Architecture.excalidraw` |
| Flow/pipeline diagram | `<Description>_Pipeline.excalidraw` | `DataIngestion_Pipeline.excalidraw` |
| Lifecycle diagram | `<Object>_Lifecycle.excalidraw` | `Session_Lifecycle.excalidraw` |
| State machine diagram | `<Object>_StateMachine.excalidraw` | `Order_StateMachine.excalidraw` |
| Class/relation diagram | `<Object>_ClassDiagram.excalidraw` | `Entity_ClassDiagram.excalidraw` |
| Memory layout diagram | `<Object>_MemoryLayout.excalidraw` | `Bucket_MemoryLayout.excalidraw` |
| Exported PNG (any type) | `<same-base-name>.png` | `Payment_Architecture.png` |

**File number uniqueness**: each file's three-digit prefix is globally unique across the entire `Document/` tree. The tens/hundreds digits identify the parent folder; the ones digit identifies the file within that folder. For example, `053_LogicModules_Analysis.md` must reside in folder `050-GameLogic/`.

**General principles:**

- Domain-specific English terms (class names, technical abbreviations) keep their original form
- Use the project's primary language for descriptive text; preserve technical terms as-is
- No spaces in file names; use underscores for English compound words
- File extension determines type: `.md` for docs, `.excalidraw` for diagrams
- Choose a consistent naming language (Chinese or English) for the project and stick with it

### Diagram Subfolder Alignment

Each `NNN-<TopicName>` topic folder MUST have a corresponding `diagrams/<ShortLabel>/` subfolder. Decide the short labels during structure analysis when the module inventory is built.

**Rules:**

- The short label should be a concise domain keyword derived from the topic folder name
- The `diagrams/Global/` subfolder holds the global architecture diagram
- Overview-level diagrams that span multiple modules can live at `diagrams/` root
- Cross-cutting or miscellaneous diagrams go into a catch-all subfolder (e.g., `diagrams/Misc/`)
- When a new topic folder is added, its corresponding diagram subfolder MUST be created immediately (even if initially empty)

### Cross-Referencing Convention

- Analysis docs reference diagrams using **relative paths** to the `.png` export: `![Architecture](../diagrams/<SubsystemLabel>/<DiagramName>.png)`
- The master index lists ALL docs and ALL diagrams with relative links
- Every analysis doc MUST be listed in the master index; orphan docs are prohibited

## Diagram Strategy

Select the diagram workflow based on whether the `excalidraw-workflow` skill is available.

### Path A: With `excalidraw-workflow` (Recommended)

1. **Generate**: Following the diagram plan (from MasterIndex Execution Plan), invoke the `excalidraw-workflow` skill to produce `.excalidraw` files into `diagrams/<DomainLabel>/`
2. **Export to PNG**: Run the export command immediately after each batch of diagrams:
   ```bash
   node <excalidraw-workflow-skill-root>/scripts/export_excalidraw_png.js ./Document/diagrams
   ```
   Incremental by default (skips already-exported files). Use `--force` to re-export after edits.
3. **Reference rule**: All `![...]()` embeds in documents MUST point to `.png`, never `.excalidraw`
4. **Index registration**: The Master Index diagrams table lists the `.png` path for viewing and notes the `.excalidraw` source exists for editing
5. **Export timing**:
   - Batch-export after all diagrams per module; run once more on the full `diagrams/` directory before writing handoff notes

### Path B: Without `excalidraw-workflow`

1. **Generate**: Write **detailed Mermaid diagrams** directly in the analysis document (use box-drawing characters, label component names and connections)
2. **Placement**: Mermaid diagrams are inline within the diagram sections of analysis documents; no separate files needed
3. **Index marking**: Mark the module's diagram status as `[mermaid-only]` in the Master Index
4. **Upgrade path**: When `excalidraw-workflow` becomes available later, Mermaid diagrams can be batch-converted to Excalidraw format

## Structure Compliance Checklist

Run this checklist at the end of each conversation to verify the Document directory remains well-structured.

### Directory Rules

- [ ] All topic folders follow `NNN-<TopicName>` naming (three-digit, zero-padded, decade-aligned)
- [ ] Topic folder numbers are multiples of 10, ordered by reading priority
- [ ] Each folder's allocated file range does not overlap with the next folder's start
- [ ] `diagrams/Global/` exists with the global architecture diagram
- [ ] `diagrams/` subfolders exist for every topic domain
- [ ] No `.excalidraw` files outside `diagrams/` or `Presentation/diagrams/`
- [ ] No `.md` analysis files outside their `NNN-` topic folder
- [ ] Root level contains only index/convention files, not analysis docs

### File Naming Rules

- [ ] Analysis docs use `NNN_` prefix + descriptive name + analysis suffix + `.md`
- [ ] File numbers are globally unique and fall within their parent folder's allocated range
- [ ] Diagram files use descriptive name + type suffix + `.excalidraw`
- [ ] No spaces in file names
- [ ] Domain-specific technical terms preserved in original form

### Cross-Reference Integrity

- [ ] Master index lists ALL analysis docs with working relative links
- [ ] Master index lists ALL diagrams with working relative links
- [ ] Analysis docs reference diagrams using relative paths (`../diagrams/...`)
- [ ] No orphan files (every file registered in master index)
- [ ] No broken links in master index
- [ ] Every `.excalidraw` file under `diagrams/` has a corresponding `.png` sibling
- [ ] All `![...]()` image embeds in analysis docs reference `.png` files, not `.excalidraw`

### Decade-Aligned Numbering Consistency

- [ ] Documentation Execution Plan includes a "Range" column for each folder's allocated file number range
- [ ] Every file number falls within its parent folder's allocated range
- [ ] No two folders have overlapping allocated ranges
- [ ] File numbers are globally unique across the entire `Document/` tree

### Directory Tree Consistency

- [ ] Every structural module in Priority Matrix is locatable in the directory tree
- [ ] Role tags (`[core]`/`[supporting]`/`[infra]`) align with Priority Matrix (P0/P1/P2)
- [ ] Status tags in tree match actual completion state (docs exist for `[done]` modules)
- [ ] `[skip]` directories correspond to SKIP or Excluded Modules entries
- [ ] `[partial]` entries specify which subset is complete
- [ ] When a module analysis doc is created, tree status updated from `[todo]` to `[done]`
- [ ] `[bottleneck]` modules in anomaly table are P0 or P1

### Feature Module Consistency

- [ ] Every constituent location in Feature Module Matrix has a `{feature:Name}` tag in the tree
- [ ] Every `{feature:Name}` tag in the tree has a corresponding Feature Module Matrix entry
- [ ] P0 feature constituent locations reside in P1-or-higher structural modules
- [ ] Feature Module Matrix lists all features detected during structure analysis
- [ ] Feature–Structure Cross-Reference table is complete and accurate
