# Codebase-Structure Skill 最佳使用模板

> 配合 `/codebase-structure` skill 使用，选择对应项目类型的模板，在触发 skill 时作为输入以获得最深度的分析和最高质量的 MasterIndex。

---

## 模板 A：通用项目深度分析（推荐）

```
/codebase-structure

请对当前项目进行 **Tier 3（Deep Analysis）** 级别的完整结构分析，输出语言为中文。

## 项目背景
- 项目类型：[填写：如 Unity SLG游戏 / Web后端服务 / 数据平台 等]
- 主要语言：[填写：如 C# / Python / TypeScript 等]
- 项目规模：[填写：如 大型 2000+ 文件 / 中型 500-2000 文件 等]
- 团队情况：[填写：如 多团队协作 / 单人项目 等]

## 已知的核心业务特性（Feature Module 提示）
以下是我已知的主要业务特性，请在特性模块检测时优先确认这些，并补充发现的遗漏特性：
1. [特性1名称]：[简要描述，如"战斗系统：包含回合制战斗逻辑、技能系统、AI决策"]
2. [特性2名称]：[简要描述]
3. [特性3名称]：[简要描述]
（如不确定可留空，让 skill 自动检测）

## 特别关注点
- [填写你特别想深入了解的架构区域]
- [填写已知的架构问题/疑惑]

## 分析要求
1. 结构模块检测时，对每个 >100 文件的目录执行强制子模块拆分
2. 特性模块检测时，执行完整的 5 步扫描策略（名称扫描+导入扫描+配置锚定+用户确认+复杂度评估）
3. 依赖分析要覆盖所有源文件，不仅是抽样
4. 标注所有 [inferred]、[boundary-uncertain] 的模块，等我确认
5. 检测并报告所有架构异常：循环依赖、瓶颈、孤立模块、层级违规
6. 为每个 P0 模块生成独立架构图
7. 为每个 P0 特性模块生成跨模块流程图
```

## 模板 B：Unity / 游戏项目专用

```
/codebase-structure

请对当前 Unity SLG 游戏项目进行 **Tier 3（Deep Analysis）** 深度结构分析，中文输出。

## 项目背景
- 项目类型：Unity SLG（策略类）手游
- 主要语言：C#（逻辑层）、可能包含 Lua/Python 热更新脚本
- 项目规模：大型项目（多个 .csproj 程序集）
- 关注重点：理解游戏架构分层、战斗系统实现、UI框架、网络通信、数据配置管理

## 已知核心业务特性（请验证并补充）
1. 战斗系统（Battle）：战斗逻辑、渲染、技能、AI
2. UI系统：界面框架、面板管理、动画
3. 网络通信：协议、同步、消息处理
4. 数据/配置管理：表格数据、配置加载、编辑器工具
5. 资源管理：AssetBundle、资源加载、热更新

## 特别关注
- Assembly（.csproj）之间的依赖关系 — 这是 Unity 项目模块化的关键边界
- Battle.Logic 与 Battle.Render 的分离模式及其交互方式
- 第三方插件（如 AstarPathfinding、DOTween 等）的集成方式和影响范围
- Editor 工具与运行时代码的边界

## 分析要求
1. 将每个 .csproj 视为一个候选结构模块边界（Unity Assembly Definition 是最强的模块信号）
2. Assets/ 目录下的子目录按功能域进行结构模块检测
3. 对所有 >100 文件的目录执行子模块拆分
4. 特性模块检测执行完整5步策略
5. 重点分析循环依赖和层级违规（Unity 项目中常见）
6. 在目录树中标注哪些是编辑器代码（Editor）、哪些是运行时代码（Runtime）
```

## 交互阶段配合指南

### 阶段 1：配置选择
- 分析深度选 **Tier 3 (Deep Analysis)**
- 语言选 **中文**

### 阶段 2：模块清单确认（最关键）

当 skill 展示模块清单请你确认时，使用以下回复模板：

```
模块清单确认：

### 结构模块
1. [模块X] 的边界正确，确认
2. [模块Y] 建议拆分为 [子模块A] 和 [子模块B]，因为它们职责完全不同
3. [模块Z] 优先级应该从 P2 提升到 P1，因为 [原因]

### 特性模块
4. 确认 [特性A]、[特性B] 的检测结果
5. 遗漏特性模块补充：
   - [特性名称]：涉及 [目录A]、[目录B]、[目录C]，核心入口是 [XXX类]
6. [特性C] 和 [特性D] 建议合并，因为它们共享 >70% 的组成位置
```

### 阶段 3：依赖图确认

```
依赖图确认：
1. 分层基本合理
2. [模块A] ↔ [模块B] 的循环依赖是已知技术债务，需要标注
3. [模块C] 确实是瓶颈模块，有 N 个下游依赖者
4. [模块D] 看起来是孤立模块，实际上它是通过反射/配置注入动态加载的
```

### 阶段 4：最终 MasterIndex 确认

```
MasterIndex 确认：
1. §50 执行计划的模块顺序合理
2. 请确认所有 P0 模块都有图表计划
3. 目录树中的 {feature:Name} 标签完整性已验证
4. 可以交付给 codebase-documentation 开始逐模块分析
```

## 质量自检清单

高质量 MasterIndex 应满足以下标准：

- [ ] 每个结构模块在 §10 矩阵和 §40 目录树中都有对应条目
- [ ] 每个特性模块的组成位置在 §40 目录树中有 `{feature:Name}` 标签
- [ ] 所有 P0 模块都有明确的证据支撑（>=3个下游依赖 或 核心业务逻辑）
- [ ] 依赖异常（循环依赖、层级违规）全部标注并说明影响
- [ ] 没有使用模糊语言（"各种"、"多个模式"、"数据流经系统"、"等"）
- [ ] 所有文件路径使用 `path/file.ext` 格式
- [ ] §50 执行计划有明确的模块顺序和预估工作量
- [ ] 特性-结构交叉引用表完整
