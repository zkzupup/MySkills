# SPA Architecture Reference

Architecture patterns for sites generated by `docs-to-interactive-site`.

## Overall Architecture

```
Document/               (markdown source files)
    |  python build_site.py
    v
site/                   (output)
  index.html            single-page application shell
  css/                  theme + layout stylesheets
  js/                   routing, render, vis-*.js modules
  data/                 all-docs.js, modules.json, search-index.json
  diagrams/             generated SVG/mermaid assets
```

All runtime behavior lives in the browser. The build script is the only
server-side step; the output is a fully static SPA that works from `file://`.

## Routing System

Hash-based routing with four route shapes:

| Pattern | View |
|---|---|
| `#/` | Home / dashboard |
| `#/module/{id}` | Module detail |
| `#/doc/{moduleId}/{docId}` | Document reader |
| `#/search?q={query}` | Search results |

Core functions:

```javascript
// Split location.hash, return { type, params }
function parseRoute() {
  var hash = location.hash.slice(2);        // strip "#/"
  var parts = hash.split('/');
  if (parts[0] === 'module') return { type: 'module', id: parts[1] };
  if (parts[0] === 'doc')    return { type: 'doc', moduleId: parts[1], docId: parts[2] };
  if (parts[0] === 'search') return { type: 'search', query: /* parse q param */ };
  return { type: 'home' };
}

function navigate() {
  var route = parseRoute();
  switch (route.type) {
    case 'home':   renderHome();                        break;
    case 'module': renderModule(route.id);              break;
    case 'doc':    renderDoc(route.moduleId, route.docId); break;
    case 'search': renderSearch(route.query);           break;
  }
}

window.addEventListener('hashchange', navigate);
```

## Dual-Mode Data Loading

**Problem:** `fetch()` fails under `file://` protocol (CORS / security restrictions).

**Solution:** The build script generates `data/all-docs.js` which sets globals:

```javascript
window.DOCS_DATA = { "mod01/intro": "<h2>...</h2>...", ... };
window.MODULES_DATA = [ { id: "010", title: "...", ... }, ... ];
window.SEARCH_INDEX_DATA = [ { title: "...", keywords: "...", ... }, ... ];
```

Loading helpers try the global first, then `fetch`:

```javascript
function loadJSON(path) {
  if (path.indexOf('modules') !== -1 && window.MODULES_DATA)
    return Promise.resolve(window.MODULES_DATA);
  if (path.indexOf('search') !== -1 && window.SEARCH_INDEX_DATA)
    return Promise.resolve(window.SEARCH_INDEX_DATA);
  return fetch(path).then(function(r) { return r.json(); });
}

function loadDocHTML(path) {
  if (window.DOCS_DATA && window.DOCS_DATA[path])
    return Promise.resolve(window.DOCS_DATA[path]);
  return fetch(path).then(function(r) { return r.text(); });
}
```

In `index.html`, the script tag uses an `onerror` fallback for HTTP mode where
the bundled JS file may not exist:

```html
<script src="data/all-docs.js" onerror="console.log('HTTP mode: loading via fetch')"></script>
```

## Visualization Registry Pattern

Each module can have a custom visualization in `js/vis-{id}.js`. The file
defines a single global function:

```javascript
// js/vis-010.js
function renderVis010(container, mod) { /* ... */ }
```

A central registry maps module IDs to render functions, checking existence
to handle missing scripts gracefully:

```javascript
function getModuleVis(id) {
  var registry = {
    '010': typeof renderVis010 !== 'undefined' ? renderVis010 : null,
    '020': typeof renderVis020 !== 'undefined' ? renderVis020 : null,
    // ...
  };
  return registry[id] || null;
}
```

`renderModule()` calls `getModuleVis(id)` first. If it returns `null`, the
fallback `renderModuleLegacy(id)` generates a generic layout.

## Theme System

CSS custom properties on `:root` (light) and `body.dark` (dark):

```css
:root {
  --bg: #ffffff;  --bg-secondary: #f5f5f5;
  --text: #1a1a1a;  --text-muted: #666666;
  --accent: #2563eb;  --border: #e0e0e0;  --shadow: rgba(0,0,0,0.08);
  /* layer colors, priority colors, etc. */
}
body.dark {
  --bg: #1a1a2e;  --bg-secondary: #16213e;
  --text: #e0e0e0;  --text-muted: #a0a0a0;
  --accent: #64b5f6;  --border: #2a2a4a;  --shadow: rgba(0,0,0,0.3);
}
```

JS toggle with localStorage persistence:

```javascript
document.body.classList.toggle('dark');
localStorage.setItem('theme', document.body.classList.contains('dark') ? 'dark' : 'light');
```

SVG elements can reference theme variables directly: `fill: 'var(--text)'`.

Mermaid.js is initialized with the matching theme:

```javascript
mermaid.initialize({ startOnLoad: false, theme: isDark ? 'dark' : 'default' });
```

## Responsive Design

Three-column grid layout:

```css
.layout { display: grid; grid-template-columns: 270px 1fr 220px; }
```

Breakpoints:

| Width | Behavior |
|---|---|
| > 1100px | Full three-column layout |
| <= 1100px | Hide TOC column, two-column |
| <= 768px | Sidebar becomes overlay, hamburger menu toggle |
| <= 480px | Compressed search, single-column stat cards |

The sidebar overlay uses `position: fixed` with a backdrop, toggled by a
hamburger button injected at the 768px breakpoint.

## Vendor Library Integration

External scripts are loaded with status tracking:

```javascript
window.__vendorStatus = {};
// In each <script> tag:
// onload="window.__vendorStatus.rough = 'loaded'"
// onerror="window.__vendorStatus.rough = 'failed'"
```

**rough.js** -- Hand-drawn style SVG rendering:

```javascript
var rc = rough.svg(svgElement);
var node = rc.rectangle(x, y, w, h, { roughness: 0.8, fill: color });
svgElement.appendChild(node);
// Fallback when rough.js unavailable: plain SVG rect/line elements
```

**mermaid.js** -- Diagram rendering, initialized lazily:

```javascript
mermaid.initialize({ startOnLoad: false });
// After injecting content with <pre class="mermaid"> blocks:
mermaid.run({ nodes: document.querySelectorAll('.mermaid') });
```

**fuse.js** -- Client-side fuzzy search:

```javascript
var fuse = new Fuse(searchIndex, {
  keys: ['title', 'keywords', 'summary'],
  threshold: 0.3
});
var results = fuse.search(query);
```

## Content Security

- **No inline scripts** in generated HTML fragments; all JS lives in `.js` files.
- **No `eval()`** or dynamic script injection at runtime.
- **All user content is pre-rendered** to HTML by the build script (markdown
  converted server-side, not in the browser).
- **Minimal XSS surface**: the only runtime user input is the search box, which
  is used as a Fuse.js query string, never inserted as raw HTML.
